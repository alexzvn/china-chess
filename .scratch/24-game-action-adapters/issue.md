# 24 — Game action adapters (toggleReady, move, resign, drawOffer, drawAccept, drawDecline)

## What to build

Create `src/server/actions/game.ts` with 6 action adapters. These are the core gameplay actions that happen during an active game.

**Adapters:**
- `handleToggleReady(ctx: RoomActionContext)` — calls `toggleReady(roomId, clientId)`, if both players ready sends `gameStart` to both + broadcasts lobby
- `handleMove(ctx: RoomActionContext)` — validates turn, calls `getLegalMoves` + `makeMove`, applies to room, sends `boardUpdate` to both players, checks check/checkmate/stalemate, sends `gameEnd` if game over
- `handleResign(ctx: RoomActionContext)` — calls `resign(roomId, clientId)`, sets room status, sends `gameEnd` to both players, broadcasts lobby
- `handleDrawOffer(ctx: RoomActionContext)` — sends `drawOffered` to opponent
- `handleDrawAccept(ctx: RoomActionContext)` — calls resign-like logic, sends `gameEnd` with result "draw" to both players, broadcasts lobby
- `handleDrawDecline(ctx: RoomActionContext)` — sends `drawDeclined` to offerer

**The `move` adapter is the most complex** (~30 lines): it validates the move through the engine, applies it, checks for check/checkmate/stalemate, and builds either a board update or a game-end notification.

## Acceptance criteria

- [ ] All 6 adapters are implemented in `src/server/actions/game.ts`
- [ ] `handleMove` validates turn, calls engine functions, handles check/checkmate/stalemate
- [ ] `handleToggleReady` sends `gameStart` only when both players are ready
- [ ] `handleResign` and `handleDrawAccept` both result in `gameEnd` but with different result types
- [ ] `handleDrawOffer` routes to opponent only (not to self)
- [ ] Error cases handled (game not active, not your turn, illegal move)
- [ ] Unit tests for each adapter
- [ ] `bun test` passes
